# IndexCache：砍掉 75% 的索引器计算，稀疏注意力再提速 1.8 倍

最近看到一篇让我眼前一亮的论文——IndexCache。

说它亮眼，不是因为又出了一个新模型，而是因为它发现了一个被忽略了很久的事实：**Transformer 相邻层的注意力模式，其实高度相似。**

这个发现看似简单，但直接带来了一个很实际的结果：在 300 亿参数的大模型上，砍掉 75% 的索引器计算，几乎不损失精度。

prefill 速度提升 1.82 倍，decode 速度提升 1.48 倍。

这不是魔法，是找到了一个一直被浪费的计算机会。

## 问题在哪？长上下文的"隐形"瓶颈

长上下文 LLM 现在已经是标配了。无论是处理长文档、多轮对话，还是 Agent 工作流，动辄就是几万甚至几十万 token 的上下文。

传统的全注意力机制，复杂度是 O(n²)。当序列长度 n 变大时，这个复杂度就成了拦路虎。

所以大家都在研究稀疏注意力。

DeepSeek 提出的 DSA（DeepSeek Sparse Attention）是其中的代表方案：用一个轻量级的"闪电索引器"，为每个 query 选择 top-k 个最相关的 token，把核心注意力的复杂度从 O(n²) 降到 O(nk)。

听起来很美好。

但有个问题：**索引器本身也是 O(n²) 的复杂度。**

而且，索引器必须在每一层独立运行。

这就好比：虽然核心注意力变快了，但每个"关卡"还是要先做一遍 O(n²) 的筛选工作。优化了一半，另一半还在。

## 关键观察：相邻层的"默契"

论文作者做了一个很简单的实验：对比 Transformer 相邻层的 top-k 选择。

结果发现，相邻层选择的 token 重合度非常高。

这其实不难理解。Transformer 的每一层都在做"渐进式理解"，而不是颠覆式重选。第 i 层觉得重要的 token，第 i+1 层大概率也觉得重要。

**既然如此，为什么不让后面的层"抄作业"？**

这就是 IndexCache 的核心思想。

## IndexCache：跨层索引复用

具体怎么做？

把所有层分成两类：

**Full 层**：老老实实跑自己的索引器，选出 top-k。

**Shared 层**：不跑索引器，直接复用最近的 Full 层的结果。

举个例子。假设你有 80 层 Transformer：

- 第 0、20、40、60 层是 Full 层，各自跑索引器
- 第 1-19 层复用第 0 层的索引结果
- 第 21-39 层复用第 20 层的索引结果
- 以此类推

这样，80 层只需要跑 4 次索引器，而不是 80 次。

**计算量直接砍掉 95%。**

当然，问题是：哪些层该当 Full 层，哪些该当 Shared 层？

论文给了两种方案。

## 方案一：Training-free —— 贪心搜索

第一种方案不需要重新训练模型。

思路很简单：在验证集上，用一个贪心算法逐层决定。

具体来说：

1. 从第 0 层开始，假设它是 Full 层
2. 逐层往后，评估：如果当前层当 Shared 层（复用前一个 Full 层），语言模型的 loss 会增加多少？
3. 如果 loss 增加超过阈值，就把当前层升级为 Full 层
4. 重复直到所有层都决定完

这种方法的好处是：**即插即用**，不需要改模型权重。

坏处是：决策是贪心的，可能不是全局最优。

## 方案二：Training-aware —— 多层蒸馏

第二种方案更精细，但需要重新训练。

思路是：让每个 Full 层的索引器"学会"服务它下面的所有 Shared 层。

具体做法：

1. 对于每个 Full 层，计算它服务的所有层的注意力分布
2. 用这些注意力分布的平均值，作为"目标"
3. 训练 Full 层的索引器，让它选出的 top-k 尽可能匹配这个目标

这样，Full 层的索引器就不再只为自己服务，而是为"整个团队"服务。

**效果更好，但代价是重新训练。**

## 实验结果：数据说话

论文在一个 30B 的 DSA 模型上做了实验。

**核心数据：**

| 指标 | 结果 |
|------|------|
| 索引器计算量减少 | 75% |
| Prefill 速度提升 | 1.82 倍 |
| Decode 速度提升 | 1.48 倍 |
| 质量损失 | 可忽略 |

更让人印象深刻的是，论文还在生产级的 GLM-5 模型上做了初步实验，结果同样积极。

这说明这个方法不只是理论上的，而是可以直接落地到生产环境。

## 我的思考

读完这篇论文，有几个点印象比较深。

**第一个是"观察"的价值。**

论文的核心贡献其实不是什么复杂的算法，而是一个简单的观察：相邻层的注意力模式相似。

这个观察一旦有了，后面的方案就水到渠成。

很多时候，一个好的研究不是"想到了什么复杂的解决方案"，而是"发现了别人没注意到的问题"。

**第二个是"复用"的力量。**

AI 系统里其实有很多"隐性冗余"。相邻层的计算、相邻 token 的表示、相似样本的处理……

IndexCache 提醒我们，找到这些冗余并利用起来，可能比堆算力更高效。

**第三个是"可落地"的重要性。**

论文不只是做了 toy experiment，而是在 30B 模型和生产模型上都验证了效果。

这种"可以直接用"的感觉，比纯理论推导有价值得多。

## 写在最后

IndexCache 解决的问题很小——只是优化了索引器的计算。

但这个问题很实际：长上下文 LLM 的推理成本，直接影响着各种 Agent 应用的可行性。

降低 75% 的索引器计算，意味着同样的硬件可以服务更多的请求，或者用更便宜的硬件达到同样的效果。

对于正在做大模型落地的团队来说，这是一个值得尝试的优化方向。
