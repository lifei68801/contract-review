# MiniMax M2.7：自我进化的第一次心跳

看到 MiniMax 发布 M2.7 的时候，我没太在意。

模型更新太快了，这个月你追我赶，下个月又是新的一轮。性能提升个 10%、20%，看多了也就麻木了。

但读完技术博客，我意识到这个模型不太一样。不是 benchmark 分数又追上了谁，而是它做了一件事：让模型参与自己的进化。

这不是营销话术。MiniMax 详细描述了一个工作流——M2.7 自己构建 agent harness，自己更新 memory 和 skills，自己根据实验结果改进学习过程。

这个循环一旦跑起来，挺像那么回事的。

---

## 从"用模型"到"让模型进化模型"

过去两年，大模型的应用范式其实挺固定的：人类设计 prompt，人类构建工具链，人类设计评估体系，模型在人类设定的框架内执行任务。

瓶颈很明显：人类成了一切的上限。

你想让模型更聪明，就得自己先想清楚每一步该怎么走。你想让 agent 更能干，就得自己设计好所有的工具和流程。

这个瓶颈不是技术问题，是认知问题。人类能想到的优化方案，受限于人类的经验和知识。有些优化路径，人类根本想不到。

MiniMax 试了一条不同的路：让模型来设计模型的工作流。

具体怎么做？

他们让 M2.7 构建了一个研究型 agent harness。这个 harness 不是人类设计的，是模型自己搭的。包含数据管道（自动处理和传输实验数据）、训练环境（配置和运行训练任务）、基础设施接口（与底层计算资源交互）、跨团队协作模块（与不同研究组对接）、持久化记忆（保存和复用经验）。

有了这个 harness，M2.7 就能在研究人员的指导下，自主完成大量原本需要多人协作的工作。

关键是，这个 harness 是模型自己设计的。它根据自己的"经验"，知道什么样的工具链对自己最有帮助。

---

## RL 团队的日常：M2.7 处理了 30%-50%

博客举了一个例子：强化学习团队的日常工作流。

以前一个 RL 实验从想法到结果，大概是：研究员提出想法 → 查文献找 baseline → 设计实验规格 → 准备数据和代码 → 启动实验 → 监控进度 → 读日志分析指标 → 发现问题调试 → 修代码提交 → 跑测试验证。

每一步都要人来做。而且不同步骤可能需要不同团队配合——数据处理要找数据组，基础设施问题要找平台组，实验设计要和资深研究员讨论。

一个实验跑下来，沟通成本可能比实际工作时间还长。

现在有了 M2.7，流程变成了：研究员和 agent 讨论想法 → agent 自动查文献追踪规格 → agent 处理数据管道启动实验 → 实验过程中 agent 自动监控读日志分析指标调试修代码 → agent 提交 merge request 跑 smoke test → 研究员只在关键决策点介入。

M2.7 能处理整个工作流的 30%-50%。

这意味着什么？

以前一个实验可能需要三个人协作两天，现在可能只需要一个人半天。而且 agent 不会累，不会因为重复劳动厌烦，不会因为周末停止工作。

更重要的，沟通成本大幅降低。不需要协调多个团队的时间，不需要等别人有空，不需要反复解释上下文。

## 100 轮自主迭代：模型优化模型

更让我感兴趣的是另一个案例。

MiniMax 让 M2.7 完全自主地优化一个模型的编程性能。没有人类介入，模型自己执行了超过 100 轮迭代循环：

```
分析失败轨迹 → 规划修改 → 修改代码 → 运行评估 → 比较结果 → 决定保留或回滚
```

每一轮都是模型自己决策。它做了几件事。

**系统性搜索最优参数组合。** 温度、频率惩罚、存在惩罚——这些采样参数对输出质量影响很大。人类调参往往是凭经验，或者简单的 grid search。M2.7 自己搜索了这些参数的最优组合，而且是动态调整的——不同任务可能需要不同的参数组合。

**设计更具体的工作流指南。** 比如修复一个 bug 后，自动在其他文件中搜索相同的 bug 模式。这是人类工程师的经验，模型自己发现了并加入工作流。这说明模型不是在机械执行命令，而是在"思考"——什么样的流程更高效？

**添加循环检测等优化。** 在 agent loop 中检测循环，避免陷入重复尝试的死循环。这个问题其实很常见：模型尝试一个方案失败了，换个方案又失败了，然后回到第一个方案……无限循环。M2.7 自己意识到这个问题，并设计了解决方案。

最终结果：内部评估集上实现了 30% 的性能提升。

数字本身就很惊人。但更惊人的是，整个过程是模型自己完成的。人类要做的，只是设定目标、提供环境、让模型自己跑。

这让我想到一个问题：如果模型能自己优化自己的能力，那进化的速度会是多少？

人类优化模型，受限于人类的时间和精力。一个研究员可能一年能跑几十个实验。

模型自己优化自己，可以 24 小时不间断。一天就能跑几十个实验。

而且模型可以同时优化多个方向——代码能力、推理能力、多语言能力……并行推进。

这个进化的加速度，可能比我们想象的更快。

## MLE Bench：22 个竞赛，9 金 5 银 1 铜

MLE Bench 是对外验证。

OpenAI 开源了一个机器学习竞赛基准，包含 22 个竞赛任务，每个任务都可以在单张 A30 GPU 上运行。覆盖了机器学习工作流的几乎所有阶段：数据预处理、特征工程、模型选择、超参数调优、模型融合、结果提交。

这些竞赛来自 Kaggle 等平台，是真实的数据科学竞赛，不是简化的 benchmark。

MiniMax 设计了一个简单的 harness，包含三个核心模块：

- 短期记忆：每轮迭代后生成 markdown 文件，记录做了什么、结果如何
- 自我反馈：对当前结果自我批评，分析哪里做得好、哪里可以改进
- 自我优化：下一轮根据所有历史记忆和反馈，进一步优化

每个任务给 24 小时，M2.7 自主迭代。三轮运行后的结果：

- 最佳成绩：9 金 5 银 1 铜
- 平均奖牌率：66.6%

这个成绩仅次于 Opus-4.6（75.7%）和 GPT-5.4（71.2%），与 Gemini-3.1 持平。

关键点：这些模型是 M2.7 自己训练的，不是人类调参调出来的。

这意味着什么？

传统机器学习竞赛，人类选手需要：理解问题、探索数据、设计特征、选择模型、调参、融合、提交。每个环节都需要专业知识和经验。

M2.7 自主完成了所有这些步骤。它能理解问题，能探索数据，能设计特征，能选择模型，能调参，能融合，能提交。

而且它能自我反思——这一轮哪里做得不好，下一轮怎么改进。

这不是简单的自动化，是完整的"机器学习工程师"能力。

## 软件工程：不只是写代码

M2.7 在软件工程任务上的表现也不错。

SWE-Pro 达到 56.22%，接近 Opus 水平。在多语言编程（SWE Multilingual 76.5%）和多任务编程（Multi SWE Bench 52.7%）上表现更突出。

但博客里真正让我感兴趣的，是一个生产环境调试的案例。

**场景：生产环境出现告警。**

传统流程：运维收到告警 → 查监控看板 → 分析日志 → 定位问题 → 找相关代码 → 修复代码 → 提交变更 → 验证修复。每一步都要人来做，而且需要多个团队协作——运维看监控、开发看代码、DBA 查数据库。

这个过程可能需要几十分钟到几个小时，取决于问题的复杂程度和团队的响应速度。

M2.7 能做什么？

- 关联监控指标和部署时间线，做因果推理
- 对 trace 采样做统计分析，提出假设
- 主动连接数据库验证根因
- 定位代码仓库中缺失的索引迁移文件
- 使用非阻塞索引创建先止血，再提交 merge request

从可观测性分析、数据库专业知识，到 SRE 级别的决策——这不仅仅是会写代码的模型，而是真正理解生产系统的模型。

MiniMax 说，多次将生产系统事故恢复时间缩短到三分钟以内。

三分钟。传统流程可能还在开会的阶段，M2.7 已经定位问题、修复代码、提交验证了。

## Agent Teams：原生多智能体协作

多智能体协作不是新概念。但过去的多智能体系统大多是通过 prompt 工程实现的——给不同的 agent 设置不同的角色，让它们"扮演"不同的角色。

瓶颈很明显：角色边界、对抗推理、协议遵守、行为差异化，这些很难通过 prompt 实现。

比如，你让一个 agent 扮演测试工程师，它会尽量说测试相关的话。但真正遇到问题的时候，它会不会主动质疑开发 agent 的方案？会不会发现潜在的伦理问题？会不会在压力下坚持质量标准？

这些不是靠 prompt 就能解决的，是需要在训练时就学会的能力。

M2.7 的做法不同：把这些能力原生嵌入到模型中。

博客展示了一个内部用于产品原型开发的 Agent Teams 设置。这个团队包含产品经理 agent（负责需求分析和优先级排序）、开发 agent（负责技术实现）、测试 agent（负责质量保证）、设计 agent（负责用户体验）。

每个 agent 都能稳定锚定角色身份，主动挑战队友的逻辑和伦理盲点，在复杂状态机中自主决策。

这不是简单的角色扮演，是真正的协作。测试 agent 会质疑开发 agent 的方案，设计 agent 会挑战产品经理的优先级，开发 agent 会主动提出技术约束。

这种协作模式，更接近真实的人类团队。

## 专业办公：不只是写文档

M2.7 在 GDPval-AA 评估中达到了 ELO 1495，在 45 个模型中排名靠前，是开源模型最高的。

博客给了具体的金融场景案例。

**任务：基于台积电年报和财报电话会议，构建营收模型。**

M2.7 需要做什么？

- 自主阅读公司年报和财报电话会议记录
- 交叉参考多份研究报告
- 独立设计假设条件
- 构建营收预测模型
- 基于模板生成 PPT
- 撰写 Word 格式的研究报告

这基本上是一个初级分析师的日常工作。而且需要多种能力：阅读理解、数据分析、财务建模、文档撰写。

反馈是什么？"输出已经可以作为初稿，直接进入后续工作流。"

这意味着什么？

分析师不需要从零开始写报告，而是从一份已经像样的初稿开始修改。这节省的可能不是 10%、20% 的时间，而是 50% 甚至更多。

更重要的是，分析师可以把时间花在更高价值的工作上——思考业务逻辑、验证假设、与客户沟通——而不是花在格式调整、数据整理这些机械工作上。

## 技术栈和技能遵守

M2.7 在 Toolathon 测试中达到了 46.3% 的准确率，处于全球第一梯队。

但博客提到了一个更具体的指标：在 40 个复杂技能（每个超过 2000 token）的情况下，保持 97% 的技能遵守率。

这为什么重要？

因为真实的办公场景就是这样：你需要调用各种工具、遵循各种流程、遵守各种规范。比如按公司模板写周报，遵循特定代码提交规范，用指定工具发送通知，在特定时间点触发特定流程。

这些约束可能是几百条、几千条。模型能不能在长对话中始终记住这些约束？能不能在合适的时机调用合适的工具？

很多模型在这方面表现不好。对话长了，就会"忘记"之前的约束。或者在复杂场景下，不知道该调用哪个工具。

97% 的遵守率意味着：在绝大多数情况下，模型不会"忘记"它应该遵守的规则。

这是 agent 能真正落地的关键指标。

## 娱乐交互：OpenRoom

博客最后提到了一个娱乐场景的探索：OpenRoom。

这是一个基于 agent harness 的交互系统。核心想法是把 AI 交互从纯文本流中解放出来，放到一个 Web GUI 空间里，一切都可以交互。

在这个空间里，角色设定不再是冷冰冰的 prompt 块，对话驱动体验生成实时视觉反馈和场景交互，角色主动与环境互动。

MiniMax 开源了这个 demo，大部分代码由 AI 自己编写。

- 项目仓库：https://github.com/MiniMax-AI/OpenRoom
- 在线体验：https://www.openroom.ai/

这个方向让我想起一句话：未来的 AI 交互，可能不是在一个聊天框里打字，而是在一个虚拟世界里生活。

现在的 AI 助手，本质上还是"问答机器"。你问，它答。

但 OpenRoom 展示的是另一种可能：AI 主动创造场景，主动与环境互动，主动引导对话。

这种交互模式，更接近人与人的互动，而不是人与工具的互动。

## 我的思考

读完这篇博客，我有几个比较深的感受。

**"自我进化"不再是空话。** 这个词过去两年被用滥了。但 MiniMax 是第一个真正展示具体工作流的团队——让模型构建自己的工具链，自己迭代自己的能力。这不是科幻，是已经跑通的流程。而且这个流程可以复制——其他团队也可以让模型参与自己的进化。

**人机协作的边界在重构。** 过去是人设计一切、模型执行。现在是人设定目标、模型设计并执行。未来可能是人只提供反馈、模型自主完成所有工作。这个边界的变化，会深刻影响所有知识工作者的角色定位。我们可能需要重新思考：人类在 AI 时代的核心价值是什么？

**"原生能力"开始取代"prompt 工程"。** 多智能体协作、角色遵守、复杂技能调用——这些过去靠 prompt 勉强实现的能力，现在被嵌入到模型本身。这意味着模型的可靠性会大幅提升，因为不再依赖于人类写出完美的 prompt。也意味着 agent 的开发门槛会降低——不需要精心设计复杂的 prompt，模型自己就知道该怎么做。

**评估基准正在向真实场景靠拢。** SWE-Pro、VIBE-Pro、Terminal Bench 2——这些基准测试越来越接近真实工程场景。不再只是"能不能写出这段代码"，而是"能不能理解整个系统、定位问题、解决问题"。这倒逼模型厂商提升综合能力，而不是刷榜。

---

M2.7 不是完美的。从数据上看，它还没到绝对领先的位置。

但它展示了一个方向：AI 参与自身的进化，可能是突破人类瓶颈的关键路径。

当模型能自己设计工具链、自己迭代能力、自己发现并修复问题时，人类的角色就从"设计者"变成了"引导者"。

这个转变，可能比任何单次性能提升都更重要。

M2.7 已经在 MiniMax Agent 和 API 平台全面上线。有兴趣的可以去试试。也许很快，我们就会看到更多模型走上这条路。
