# MiniMax M2.7：自我进化的第一次心跳

前两天看到 MiniMax 发布 M2.7 的时候，我并没有太在意。

毕竟现在模型更新太快了，这个月你追我赶，下个月又是新的一轮。性能提升个 10%、20%，好像也不是什么稀奇事。

但仔细读完他们发的技术博客后，我意识到这个模型不太一样。

不是因为它的 benchmark 分数又追上了谁，而是因为它做了一件事：**让模型参与自己的进化。**

这不是营销话术。MiniMax 在博客里详细描述了一个工作流：让 M2.7 自己构建 agent harness，自己更新 memory 和 skills，自己根据实验结果改进学习过程。

这个循环一旦跑起来，就是自我进化的雏形。

---

## 一、从"用模型"到"让模型进化模型"

过去两年，大模型的应用范式其实挺固定的：

1. 人类设计 prompt
2. 人类构建工具链
3. 人类设计评估体系
4. 模型在人类设定的框架内执行任务

这个范式的瓶颈很明显：人类成为了一切的上限。

你想让模型更聪明，就得自己先想清楚每一步该怎么走。你想让 agent 更能干，就得自己设计好所有的工具和流程。

但 MiniMax 在 M2.7 上尝试了一条不同的路：**让模型来设计模型的工作流。**

具体怎么做？

他们首先让 M2.7 构建了一个研究型 agent harness。这个 harness 不是人类设计的，而是模型自己搭建的。它包含：

- **数据管道**：自动处理和传输实验数据
- **训练环境**：配置和运行训练任务
- **基础设施接口**：与底层计算资源交互
- **跨团队协作模块**：与不同研究组对接
- **持久化记忆**：保存和复用经验

有了这个 harness，M2.7 就能在研究人员的指导下，自主完成大量原本需要多人协作的工作。

## 二、RL 团队的日常：M2.7 处理了 30%-50%

博客里举了一个具体的例子：强化学习团队的日常工作流。

以前，一个 RL 实验从想法到结果，大概是这样：

1. 研究员提出实验想法
2. 查文献、找 baseline
3. 设计实验规格
4. 准备数据和代码
5. 启动实验
6. 监控实验进度
7. 读日志、分析指标
8. 发现问题、调试
9. 修代码、提交
10. 跑测试、验证结果

这十个步骤，每一步都要人来做。而且不同步骤可能需要不同团队的人配合——数据处理要找数据组，基础设施问题要找平台组。

现在有了 M2.7，流程变成了这样：

1. 研究员和 agent 讨论实验想法
2. Agent 自动查文献、追踪实验规格
3. Agent 自动处理数据管道、启动实验
4. 实验过程中，agent 自动监控进度
5. Agent 自动读日志、分析指标、调试、修代码
6. Agent 自动提交 merge request、跑 smoke test
7. 研究员只在关键决策点介入

**M2.7 能处理整个工作流的 30%-50%。**

这意味着什么？

以前一个实验可能需要三个人协作两天，现在可能只需要一个人半天。

而且，agent 不会累，不会因为重复劳动而厌烦，不会因为周末而停止工作。

## 三、100 轮自主迭代：模型优化模型

更让我印象深刻的，是另一个案例。

MiniMax 让 M2.7 完全自主地优化一个模型的编程性能。没有人类介入，模型自己执行了超过 100 轮迭代循环：

```
分析失败轨迹 → 规划修改 → 修改代码 → 运行评估 → 比较结果 → 决定保留或回滚
```

每一轮都是模型自己决策。它要做的事情包括：

**第一，系统性搜索最优参数组合。**

温度（temperature）、频率惩罚（frequency penalty）、存在惩罚（presence penalty）——这些采样参数对模型输出质量影响很大。M2.7 自己搜索了这些参数的最优组合。

**第二，设计更具体的工作流指南。**

比如，修复一个 bug 后，自动在其他文件中搜索相同的 bug 模式。这是人类工程师的经验，模型自己发现了并加入工作流。

**第三，添加循环检测等优化。**

在 agent loop 中检测循环，避免模型陷入重复尝试的死循环。

最终结果：**在内部评估集上实现了 30% 的性能提升。**

这个数字本身就很惊人。但更惊人的是，整个过程是模型自己完成的。

人类要做的，只是设定目标、提供环境、让模型自己跑。

## 四、MLE Bench：22 个竞赛，9 金 5 银 1 铜

如果说上面的案例是内部实验，那 MLE Bench 就是对外验证。

OpenAI 开源了一个机器学习竞赛基准测试，包含 22 个竞赛任务，每个任务都可以在单张 A30 GPU 上运行。这些竞赛覆盖了机器学习工作流的几乎所有阶段：

- 数据预处理
- 特征工程
- 模型选择
- 超参数调优
- 模型融合
- 结果提交

MiniMax 设计了一个简单的 harness，包含三个核心模块：

**短期记忆**：每一轮迭代后生成一个 markdown 文件，记录做了什么、结果如何。

**自我反馈**：对当前轮次的结果进行自我批评，分析哪里做得好、哪里可以改进。

**自我优化**：下一轮根据所有历史记忆和反馈，进行进一步优化。

每个任务给 24 小时，M2.7 自主迭代。三轮运行后的结果：

- **最佳成绩**：9 金 5 银 1 铜
- **平均奖牌率**：66.6%

这个成绩仅次于 Opus-4.6（75.7%）和 GPT-5.4（71.2%），与 Gemini-3.1（66.6%）持平。

**关键点在于：这些模型是 M2.7 自己训练的，不是人类调参调出来的。**

## 五、软件工程：不只是写代码

M2.7 在软件工程任务上的表现也很亮眼。

SWE-Pro 达到 56.22%，接近 Opus 的最好水平。在多语言编程（SWE Multilingual 76.5%）和多任务编程（Multi SWE Bench 52.7%）上表现更突出。

但博客里真正让我感兴趣的，是一个生产环境调试的案例。

**场景：生产环境出现告警。**

传统流程：

1. 运维收到告警
2. 查监控看板
3. 分析日志
4. 定位问题
5. 找相关代码
6. 修复代码
7. 提交变更
8. 验证修复

每一步都要人来做，而且需要多个团队协作——运维看监控、开发看代码、DBA 查数据库。

M2.7 能做什么？

- 关联监控指标和部署时间线，做因果推理
- 对 trace 采样做统计分析，提出假设
- 主动连接数据库验证根因
- 定位代码仓库中缺失的索引迁移文件
- 使用非阻塞索引创建先止血，再提交 merge request

**从可观测性分析、数据库专业知识，到 SRE 级别的决策——这不仅仅是会写代码的模型，而是真正理解生产系统的模型。**

MiniMax 说，多次将生产系统事故恢复时间缩短到三分钟以内。

## 六、Agent Teams：原生多智能体协作

多智能体协作（multi-agent）不是新概念。但你可能注意到了，过去的多智能体系统大多是通过 prompt 工程实现的——给不同的 agent 设置不同的角色，让它们"扮演"不同的角色。

这种方式的瓶颈很明显：**角色边界、对抗推理、协议遵守、行为差异化，这些很难通过 prompt 实现。**

M2.7 的做法不同：**把这些能力原生地嵌入到模型中。**

博客里展示了一个内部用于产品原型开发的 Agent Teams 设置。这个团队包含一个最小化的组织架构：

- 产品经理 agent：负责需求分析和优先级排序
- 开发 agent：负责技术实现
- 测试 agent：负责质量保证
- 设计 agent：负责用户体验

每个 agent 都能：

- **稳定锚定角色身份**：不会在对话中"忘记"自己是谁
- **主动挑战队友的逻辑和伦理盲点**：比如测试 agent 质疑开发 agent 的方案是否有安全隐患
- **在复杂状态机中自主决策**：不是简单的 if-else，而是根据上下文做出判断

这需要模型在训练时就学会协作，而不是事后靠 prompt 来"弥补"。

## 七、专业办公：不只是写文档

M2.7 在 GDPval-AA 评估中达到了 ELO 1495，在 45 个模型中排名靠前，是开源模型中最高的。

但分数只是数字，博客里给了具体的金融场景案例。

**任务：基于台积电年报和财报电话会议，构建营收模型。**

M2.7 需要做什么？

1. 自主阅读公司年报和财报电话会议记录
2. 交叉参考多份研究报告
3. 独立设计假设条件
4. 构建营收预测模型
5. 基于模板生成 PPT
6. 撰写 Word 格式的研究报告

这基本上是一个初级分析师的日常工作。

反馈是什么？**"输出已经可以作为初稿，直接进入后续工作流。"**

这意味着什么？

分析师不需要从零开始写报告，而是从一份已经像样的初稿开始修改。这节省的不是 10%、20% 的时间，可能是 50% 甚至更多。

## 八、技术栈和技能遵守

M2.7 在 Toolathon 测试中达到了 46.3% 的准确率，处于全球第一梯队。

但博客里提到了一个更具体的指标：**在 40 个复杂技能（每个超过 2000 token）的情况下，保持 97% 的技能遵守率。**

这为什么重要？

因为真实的办公场景就是这样：你需要调用各种工具、遵循各种流程、遵守各种规范。

比如，你可能需要：

- 按照公司模板写周报
- 遵循特定的代码提交规范
- 使用指定的工具发送通知
- 在特定的时间点触发特定的流程

这些约束可能是几百条、几千条。模型能不能在长对话中始终记住这些约束？能不能在合适的时机调用合适的工具？

97% 的遵守率意味着：在绝大多数情况下，模型不会"忘记"它应该遵守的规则。

## 九、娱乐交互：OpenRoom

博客最后提到了一个娱乐场景的探索：OpenRoom。

这是一个基于 agent harness 的交互系统。核心想法是：**把 AI 交互从纯文本流中解放出来，放到一个 Web GUI 空间里，一切都可以交互。**

在这个空间里：

- 角色设定不再是冷冰冰的 prompt 块
- 对话驱动体验，生成实时视觉反馈和场景交互
- 角色主动与环境互动

MiniMax 开源了这个 demo，大部分代码由 AI 自己编写。

项目仓库：https://github.com/MiniMax-AI/OpenRoom

在线体验：https://www.openroom.ai/

这个方向让我想起了一句话：**未来的 AI 交互，可能不是在一个聊天框里打字，而是在一个虚拟世界里生活。**

## 十、我的思考

读完这篇博客，我有几个比较深的感受。

**第一，"自我进化"不再是空话。**

这个词过去两年被用滥了。但 MiniMax 是第一个真正展示具体工作流的团队——让模型构建自己的工具链，自己迭代自己的能力。

这不是科幻，是已经跑通的流程。

**第二，人机协作的边界在重构。**

过去是人设计一切、模型执行。现在是人设定目标、模型设计并执行。未来可能是人只提供反馈、模型自主完成所有工作。

这个边界的变化，会深刻影响所有知识工作者的角色定位。

**第三，"原生能力"开始取代"prompt 工程"。**

多智能体协作、角色遵守、复杂技能调用——这些过去靠 prompt 勉强实现的能力，现在被嵌入到模型本身。

这意味着模型的可靠性会大幅提升，因为不再依赖于人类写出完美的 prompt。

**第四，评估基准正在向真实场景靠拢。**

SWE-Pro、VIBE-Pro、Terminal Bench 2——这些基准测试越来越接近真实工程场景。不再只是"能不能写出这段代码"，而是"能不能理解整个系统、定位问题、解决问题"。

这倒逼模型厂商提升综合能力，而不是刷榜。

## 写在最后

M2.7 不是完美的。从数据上看，它还没到绝对领先的位置。

但它展示了一个方向：**AI 参与自身的进化，可能是突破人类瓶颈的关键路径。**

当模型能自己设计工具链、自己迭代能力、自己发现并修复问题时，人类的角色就从"设计者"变成了"引导者"。

这个转变，可能比任何单次性能提升都更重要。

MiniMax 说，M2.7 已经在 MiniMax Agent 和 API 平台全面上线。有兴趣的可以去试试。

也许很快，我们就会看到更多模型走上这条路。
