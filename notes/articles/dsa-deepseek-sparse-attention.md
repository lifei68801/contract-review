# DeepSeek DSA：给稀疏注意力派打的一针强心剂

在上篇文章《从Delta Rule到DPLR优化，详解Kimi Linear的线性注意力机制》中，我们系统梳理了高效注意力的四大门派：

- **全注意力派**：死磕工程优化，FlashAttention、PagedAttention、Ring Attention 轮番上阵
- **稀疏注意力派**：只看一部分，Longformer、滑动窗口各显神通
- **纯线性派**：降维打击，Mamba、RWKV 用 O(1) 显存换速度
- **混合架构派**：缝合怪，Kimi Linear 用 3:1 配比打组合拳

文章发布后，有朋友问我：**稀疏注意力派的"中间遗忘"问题，真的无解吗？**

今天要讲的 DeepSeek DSA（DeepSeek Sparse Attention），就是 DeepSeek 团队给出的答案。

---

## 一、稀疏注意力派的"阿喀琉斯之踵"

先回顾一下稀疏注意力派的核心问题。

### 1.1 传统方案的困境

上篇文章提到，稀疏注意力派的做法是"既然全看一遍太贵，那我就只看一部分"：

- **滑动窗口（SWA）**：模型每次只和前后窗口内的词计算注意力
- **稀疏采样**：每隔几个词抽样看一次

这种做法保住了局部的理解力，但有个致命伤：**Lost in the Middle（中间遗忘）**。

当模型需要跨越几十万字进行关联推理时，中间的关键信息很容易被遗漏。模型变成了一个"只顾眼前的近视眼"。

### 1.2 问题根源

为什么会这样？

传统稀疏方案的"选择策略"是**人工设计**的——要么固定窗口，要么按固定间隔采样。但语言的结构是动态的：

- 一个代词应该指代哪个名词？
- 一个结论应该关联哪个论据？
- 一段话应该回溯哪个伏笔？

这些问题的答案，**取决于上下文**，而不是位置。

**人工规则无法穷尽所有情况**。这就是传统稀疏注意力的死结。

---

## 二、DSA 的破局思路：让模型自己学会"看"

### 2.1 核心创新

DeepSeek 团队提出了一个很直接的想法：

> **与其人工设计规则，不如让模型自己学习"哪些位置重要"。**

DSA（DeepSeek Sparse Attention）引入了一个轻量级的**全局"闪电索引器"（Lightning Indexer）**，动态筛选每个 Query 应该关注的 Top-k 个 Key。

这就像是给模型配了一副"智能眼镜"——不是固定看某个方向，而是根据当前内容，自动聚焦到最相关的位置。

### 2.2 一句话概括

**用一个轻量级的索引器快速打分，找出每个 Query 应该关注的 Top-k 个 Key，然后只对这些 Key 做精确的注意力计算。**

用一个类比：

- **传统注意力**：把图书馆所有书都翻一遍（O(L²)）
- **传统稀疏注意力**：只看固定的几排书架（容易漏）
- **DSA**：先查索引卡，找到最相关的几排书架，再仔细翻（精准且高效）

---

## 三、技术细节：两个核心组件

DSA 由两个核心组件构成：

### 3.1 Lightning Indexer（闪电索引器）

这是 DSA 的"索引卡"——一个轻量级的打分机制，用来快速判断哪些 token 之间可能存在重要关联。

对于第 t 个 Query token $\mathbf{h}_t$ 和第 s 个前序 token $\mathbf{h}_s$，索引分数计算如下：

$$
I_{t,s} = \sum_{j=1}^{H^I} w_{t,j}^I \cdot \text{ReLU}\left(\mathbf{q}_{t,j}^I \cdot \mathbf{k}_s^I\right)
$$

**逐符号解释：**

| 符号 | 含义 | 说明 |
|------|------|------|
| $H^I$ | 索引器的头数 | 数量很小，计算效率高 |
| $\mathbf{q}_{t,j}^I$ | Query 派生的查询向量 | 从当前 token 提取 |
| $\mathbf{k}_s^I$ | 前序 token 的键向量 | 从历史 token 提取 |
| $w_{t,j}^I$ | 每个头的权重 | 学习不同头的重要性 |
| **ReLU** | 激活函数 | 非负、计算简单 |

**为什么用 ReLU 而不是 Softmax？**

传统注意力用 Softmax 归一化，但 DSA 的索引器不需要归一化——它的任务只是**排序**，找出分数最高的 Top-k 个位置。ReLU 非负、计算简单，完全够用。

### 3.2 Fine-grained Token Selection（细粒度 Token 选择）

有了索引分数 $\{I_{t,s}\}$，下一步就是选择：

$$
\mathbf{u}_t = \text{Attn}\left(\mathbf{h}_t, \{\mathbf{c}_s | I_{t,s} \in \text{Top-k}(I_{t,:})\}\right)
$$

**通俗解释：**

对于每个 Query token，只保留索引分数最高的 k 个 Key-Value 对，然后对这些被选中的对做**精确的**注意力计算。

论文中 k 设置为 **2048**。这意味着无论序列多长，每个 Query 最多只关注 2048 个位置。

### 3.3 在 MLA 框架下的实现

DeepSeek-V3.2 使用的是 MLA（Multi-Head Latent Attention）架构——上篇文章提到的"把 KV Cache 打成白菜价"的利器。

DSA 基于 MLA 的 **MQA 模式**实现：

- **MHA 模式**：每个 head 有独立的 KV cache
- **MQA 模式**：所有 query heads 共享同一个 latent vector 作为 KV

DSA 选择 MQA 模式，原因是**计算效率**——在 kernel 层面，每个 KV entry 需要被多个 Query 共享，才能充分利用 GPU 的并行能力。

## 四、训练过程：两阶段策略

DSA 不能直接拿来用，需要经过两阶段训练才能让模型适应稀疏注意力模式。

### 4.1 Dense Warm-up Stage（密集预热阶段）

**目标**：初始化 Lightning Indexer，让它学会模仿原来密集注意力的行为。

**具体做法：**

1. **冻结所有模型参数**，只训练 Lightning Indexer
2. **保持密集注意力**，不引入稀疏选择
3. **对齐目标**：让索引器的输出分布接近主注意力的分布

具体来说，对于第 t 个 Query，先聚合所有 attention heads 的注意力分数，得到一个目标分布 $p_{t,:}$，然后用 KL 散度训练索引器：

$$
\mathcal{L}^I = \sum_t \mathbb{D}_{\text{KL}}\left(p_{t,:} \| \text{Softmax}(I_{t,:})\right)
$$

**这个阶段非常短**：只有 1000 步，每步 16 个 128K 序列，总共约 **21 亿 token**。

### 4.2 Sparse Training Stage（稀疏训练阶段）

**目标**：让整个模型适应稀疏注意力模式。

**具体做法：**

1. **引入稀疏选择**：只保留 Top-k 的 KV entries
2. **优化所有参数**：包括索引器和主模型
3. **继续对齐索引器**：但只对被选中的 token 计算损失

损失函数改为：

$$
\mathcal{L}^I = \sum_t \mathbb{D}_{\text{KL}}\left(p_{t,\mathcal{S}_t} \| \text{Softmax}(I_{t,\mathcal{S}_t})\right)
$$

其中 $\mathcal{S}_t = \{s | I_{t,s} \in \text{Top-k}(I_{t,:})\}$ 是被选中的 token 集合。

**关键技术细节：**

| 参数 | 值 |
|------|-----|
| 学习率 | 7.3×10⁻⁶ |
| 选择数量 k | 2048 |
| 训练步数 | 15000 步 |
| 每步序列 | 480 个 128K 序列 |
| 总 token 量 | 约 9437 亿 |

## 五、性能表现：解决了"中间遗忘"吗？

### 5.1 复杂度分析

| 组件 | 复杂度 | 说明 |
|------|--------|------|
| 主注意力（MLA） | O(L²) → O(Lk) | k = 2048 |
| Lightning Indexer | O(L²) | 但计算量远小于 MLA |

读者可能会问：Lightning Indexer 不是还是 O(L²) 吗？

是的，但它的计算量远小于 MLA：
- 头数很少（$H^I$ 小）
- 可以用 **FP8** 实现
- ReLU 激活比 Softmax 快得多

### 5.2 实际推理成本

论文给出了在 H800 GPU 上的实测数据：

| 阶段 | DeepSeek-V3.1 | DeepSeek-V3.2 (DSA) |
|------|---------------|---------------------|
| Prefilling | 成本随 L² 增长 | 成本线性增长 |
| Decoding | 标准速度 | 128K 时提速 **50%** |

### 5.3 模型性能：没有退化！

这是最关键的问题：稀疏注意力会不会损害模型性能？

论文从三个维度验证了 DSA 的性能：

**1. 标准基准测试**

DeepSeek-V3.2-Exp 与 DeepSeek-V3.1-Terminus 在各类 benchmark 上表现相近，无论短上下文还是长上下文任务。

**2. 人类偏好（Chatbot Arena）**

两个模型的 Elo 分数几乎一致。DSA 的引入没有影响对话质量。

**3. 长上下文评测**

在独立的长上下文评测中（AA-LCR、Fiction.liveBench），V3.2-Exp 甚至**略优于** V3.1-Terminus。

**这说明了什么？**

DSA 的细粒度稀疏选择机制，有效地保留了关键信息的传递。模型学会了"看哪里重要"，不需要遍历所有位置。

传统稀疏注意力的"中间遗忘"问题，被 DSA 用**智能索引**解决了。

## 六、对比：四门派的技术选择

写到这里，我们来对比一下四条路线的技术选择：

### 6.1 核心思路对比

| 门派 | 核心思路 | 复杂度 | 精度 | 工程难度 |
|------|----------|--------|------|----------|
| 全注意力派 | 工程优化 | O(L²) | 最高 | 中 |
| 传统稀疏派 | 固定规则选择 | O(Lk) | 较低 | 低 |
| **DSA** | **智能索引选择** | **O(Lk)** | **高** | **高** |
| 纯线性派 | 数学近似 | O(L) | 较低 | 高 |
| 混合架构派 | 线性+全注意力 | 混合 | 高 | 高 |

### 6.2 适用场景

| 场景 | 推荐方案 |
|------|----------|
| 极致速度要求、精度可妥协 | 纯线性派（Mamba、RWKV） |
| 长上下文推理、高精度要求 | DSA 或 混合架构 |
| 流式处理、增量计算 | 纯线性派 或 混合架构 |
| 短文本、资源充足 | 全注意力派 |

### 6.3 我的判断

四条路线不是非此即彼的关系，未来很可能是**混合架构**：

- **局部**：用线性注意力（效率）
- **全局**：用 DSA 稀疏注意力（精度）
- **关键节点**：用全注意力（兜底）

Kimi Linear 的 3:1 配比是一个方向，但也许会有更灵活的动态路由机制——根据文本特征自动选择注意力类型。

---

## 七、写在最后

从 Kimi Linear 到 DeepSeek DSA，我们看到了高效注意力领域的两条重要路线：

- **混合架构派**：用线性+全注意力的组合，追求速度与精度的平衡
- **智能稀疏派**：用学习型索引器，让模型自己学会"看哪里重要"

两条路线都回答了同一个问题：**如何在保持精度的前提下，突破 O(n²) 的限制？**

DSA 给稀疏注意力派打了一针强心剂——它证明了，稀疏注意力不一定意味着"中间遗忘"，关键在于选择策略是否够智能。

未来会怎样？

也许纯线性模型会找到更精妙的压缩机制，也许混合架构会更加成熟，也许 DSA 的思想会被更多架构借鉴。

无论如何，这个领域的创新速度，让人兴奋。
